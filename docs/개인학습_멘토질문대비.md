## 1. Git 충돌 해결 (Conflict Resolution)

### 핵심 개념
- `git stash pop` 후 두 종류의 충돌 발생
  - **UU (both modified)**: 양쪽이 같은 파일을 수정 → 수동 머지
  - **DU (deleted by us)**: 우리 브랜치에서 삭제됐는데 stash에 수정본 있음 → `git rm`으로 해결
- 충돌 해결 원칙: dev의 UI 변경(CSS variables) + 내 기능(onClick, cursor-pointer) 둘 다 살림

### 예상 질문
**Q. git stash pop 충돌이 났을 때 어떻게 처리했나요?**
> KanbanCard.tsx는 dev가 CSS 변수로 바꾼 스타일과 내가 추가한 onClick/cursor-pointer 로직이 충돌했어요. `<<<<<<< Updated upstream` 이 dev 코드고 `>>>>>>> Stashed changes`가 내 코드였는데, 두 부분을 합쳐서 dev의 `var(--kb-border)` CSS 변수는 유지하고 내 cursor-pointer 조건부 클래스도 살렸어요. ChatOverlay.tsx는 dev에서 삭제된 파일인데 stash에 수정본이 있었으니 `git rm`으로 conflict 해소했어요.

**Q. rebase vs merge 차이는?**
> merge는 병합 커밋이 생기고, rebase는 히스토리를 직선으로 유지해요. 저희 팀은 `git pull origin dev --rebase`를 쓰는데, 이러면 내 커밋이 dev 최신 위에 올라가서 히스토리가 깔끔해져요.

## 2. React 이벤트 버스 (CustomEvent)

### 핵심 개념
```
window.dispatchEvent(new CustomEvent("boss:artifacts-changed"))
window.addEventListener("boss:artifacts-changed", handler)
```
- 서로 다른 컴포넌트(InlineChat ↔ KanbanBoard) 가 부모-자식 관계가 아닐 때 상태 동기화
- KanbanBoard가 `boss:artifacts-changed`를 **듣지 않아서** 저장 후 자동 새로고침이 안 됐음
- useEffect cleanup에서 `removeEventListener` 필수 (메모리 누수 방지)

### 예상 질문
**Q. InlineChat에서 저장하면 왜 KanbanBoard가 자동으로 새로고침이 안 됐나요?**
> KanbanBoard는 마운트 시 한 번만 `load()`를 호출했어요. `boss:artifacts-changed` 이벤트를 발행은 했지만 구독하지 않아서 외부 저장 이벤트를 감지 못했어요. `useEffect`에 이벤트 리스너를 등록하고 cleanup에서 제거하는 로직을 추가해서 해결했어요.

**Q. props drilling 대신 CustomEvent를 쓴 이유는?**
> InlineChat과 KanbanBoard는 공통 부모가 없고, Context로 연결하기엔 scope가 달라요(대시보드 vs 도메인 페이지). 전역 이벤트 버스가 가장 간단한 해법이에요. Redux나 Zustand 같은 상태관리 라이브러리를 쓸 수도 있지만, 단순한 "새로고침 트리거"엔 오버스펙이에요.

## 3. 모달 인라인화 & 상태 설계

### 핵심 개념
- 기존: `boss:focus-node` → 캔버스 이동 (캔버스가 삭제된 상태라 동작 안 함)
- 개선: `savedArtifactMeta` 상태를 Message 타입에 추가 → 저장 시 `{ type, recordedDate, title }` 저장 → 클릭 시 SalesDetailModal 인라인 오픈
- 핵심 패턴: **저장 시 메타데이터를 함께 저장** → 나중에 모달 열 때 별도 API 호출 없이 사용

### 예상 질문
**Q. SalesDetailModal을 InlineChat 안에서 여는 방법이 뭔가요?**
> `detailModal` state를 InlineChat에 추가했어요. 저장 성공 시 `savedArtifactMeta: { type, recordedDate, title }`를 메시지에 저장하고, "상세 보기" 버튼 클릭 시 `setDetailModal(...)`로 상태를 세팅해요. SalesDetailModal은 `detailModal && userId` 조건에서만 렌더링돼요.

**Q. 모달이 열릴 때 데이터를 어떻게 로드하나요?**
> SalesDetailModal 내부에서 `useCallback` + `useEffect`로 `open`이 true가 될 때 `/api/sales?start_date=X&end_date=X`를 호출해요. `recordedDate` prop이 날짜 범위로 쓰여서, 그날 저장된 모든 records가 조회돼요.

## 4. DB 정렬 & Limit 버그

### 핵심 개념
- **문제**: `ORDER BY recorded_date DESC`만 있으면 같은 날짜 내 순서가 보장 안 됨
- **결과**: 오늘 228개 레코드 → `limit=200` 초과 → 최신 저장 데이터가 잘림
- **해결**: `ORDER BY created_at DESC`로 변경 + limit 500으로 상향

```python
# Before
.order("recorded_date", desc=True).limit(200)

# After  
.order("created_at", desc=True).limit(500)
```

### 예상 질문
**Q. 왜 새로 저장한 데이터가 모달에서 안 보였나요?**
> DB에 228개 레코드가 있었고, 모달이 `limit=200`으로 조회했어요. 정렬이 `recorded_date DESC`인데 오늘 날짜 레코드는 전부 같은 날짜라 내부 순서가 불확정이에요. 결국 최신 데이터가 200개 밖으로 밀려났어요. `created_at DESC`로 정렬하면 최신이 항상 앞으로 오기 때문에 이 문제가 해결돼요.

**Q. PostgreSQL에서 ORDER BY 없이 결과 순서를 보장할 수 있나요?**
> 없어요. 테이블 스캔 순서는 vacuum, 인덱스, 병렬 쿼리 등에 따라 달라져요. 반드시 명시적 ORDER BY가 있어야 해요. 특히 같은 값이 많은 컬럼 하나로만 정렬하면 같은 값 내에서 순서가 비결정적이에요.

## 5. 영수증 OCR 파이프라인

### 전체 흐름
```
이미지 업로드
  → /api/uploads/document (artifact 저장, "receipt" 분류)
  → InlineChat이 receipt 감지
  → /api/sales/ocr (GPT-4o Vision으로 항목 파싱)
  → salesAction 마커 → 💾 저장 버튼
  → /api/sales (sales_records + revenue_entry artifact)
  → 📋 상세 보기 → SalesDetailModal
```

### 핵심 개념
- **GPT-4o Vision**: 이미지를 base64 인코딩 후 OpenAI API에 전송, JSON 형식으로 항목 추출
- **`response_format: json_object`**: GPT 응답을 강제로 JSON으로 받기 위한 옵션. 이때 prompt에 반드시 JSON 구조 명시, raw array 반환 불가 → `{"items":[...]}` 래퍼 필요
- **파일 분류 이중화**: 업로드 시 문서 저장(artifact) + OCR 분석(records) 두 가지 동시 수행

### 예상 질문
**Q. 영수증 이미지에서 매출 데이터를 어떻게 추출하나요?**
> `/api/sales/ocr` 엔드포인트에서 이미지를 base64로 인코딩해서 GPT-4o Vision API에 보내요. 시스템 프롬프트에 "item_name, category, quantity, unit_price를 JSON으로 반환하라"고 지시하고, `response_format: json_object`로 구조화된 응답을 받아요. 파싱 실패 시 `{"items":[]}`를 반환하도록 지시해서 오류 처리해요.

**Q. OCR 결과가 틀릴 수 있는데 어떻게 처리하나요?**
> OCR 결과를 바로 저장하지 않고 SalesInputTable(편집 가능한 표)로 먼저 보여줘요. 사용자가 내용을 확인/수정 후 저장 버튼을 눌러야 실제로 DB에 저장돼요. LLM 기반 파싱은 항상 검토 단계를 거치는 설계를 해요.

## 6. Sales 데이터 아키텍처

### 두 개의 테이블 역할
| 테이블 | 역할 |
|--------|------|
| `sales_records` | 실제 매출 데이터 (item_name, quantity, unit_price, amount) |
| `artifacts` (type=revenue_entry) | 캔버스/칸반 UI용 노드. metadata에 recorded_date, total_amount |

- `revenue_entry` artifact → `Reports` 서브허브에 연결 (`artifact_edges`)
- `sales_records`는 날짜 기반 조회, `artifact`는 칸반 카드로 표시

### 예상 질문
**Q. sales_records와 artifacts를 왜 분리했나요?**
> `sales_records`는 원시 데이터 분석용(집계, 리포트, RAG 검색)이고, `artifacts`는 UI 노드 역할이에요. 하나의 revenue_entry artifact가 여러 sales_records를 묶는 구조예요. 칸반에선 artifact 단위로 보여주고, 상세 모달에선 해당 날짜의 sales_records를 조회해요.

**Q. artifact_edges는 왜 필요한가요?**
> DAG(유향 비순환 그래프) 구조로 노드 간 관계를 표현해요. `parent_id=Reports서브허브, child_id=revenue_entry artifact, relation=contains` 형태로 저장하면, 칸반이 "Reports 컬럼 = Reports 서브허브의 자식 artifact들"로 조회할 수 있어요.

## 7. FastAPI + Pydantic 검증

### 핵심 개념
```python
class SaleItemIn(BaseModel):
    amount: int              # required (default 없음)
    quantity: int = 1        # optional (default 있음)
    
    @field_validator("quantity")
    @classmethod
    def validate_quantity(cls, v: int) -> int:
        if v <= 0:
            raise ValueError("quantity must be > 0")
        return v
```
- Pydantic v2: `@field_validator` + `@classmethod` 필수
- 필수 필드 누락 → 422 Unprocessable Entity 자동 반환
- 프론트에서 `amount: quantity * unit_price` 계산 후 전송

### 예상 질문
**Q. 프론트에서 amount를 계산해서 보내는 이유가 뭔가요?**
> 백엔드 `SaleItemIn`에 `amount`가 required field라서요. 프론트가 `quantity * unit_price`를 계산해서 전송해요. 단가를 모를 땐 `unit_price=0`이라 `amount=0`이 돼요. 실제론 amount를 백엔드에서 계산하고 front에서 단가/수량만 보내는 게 더 안전한 설계예요(프론트 조작 가능성 차단).
